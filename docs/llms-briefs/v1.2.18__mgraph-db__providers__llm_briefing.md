## Provider Overview

MGraph-DB includes 8 specialized providers, each optimized for different graph modeling scenarios:

| Provider | Primary Use Case | Key Features |
|----------|-----------------|--------------|
| **Simple** | Basic graphs, prototyping | Minimal schema, flexible node/edge attributes |
| **JSON** | JSON data structures | Preserves JSON hierarchy, supports arrays/objects |
| **Mermaid** | Diagram generation | Visual graph representation, multiple diagram types |
| **File System** | Directory structures | Hierarchical folders, path navigation |
| **Time Series** | Temporal data points | Precise timestamps, timezone support |
| **Time Chain** | Time-based hierarchies | Yearâ†’Monthâ†’Dayâ†’Hour chains |
| **Graph RAG** | LLM knowledge graphs | Entity extraction, relationship mapping |
| **RSS** | Feed processing | RSS/Atom feed structures |

## Provider-Specific Gotchas

### JSON Provider
- Property nodes are separate from value nodes
- Arrays maintain order through edge ordering
- Deep nesting creates many intermediate nodes

### Mermaid Provider
- Node keys must be unique (use `Safe_Id`)
- Some shapes require specific syntax
- Markdown in labels needs special handling

### File System Provider
- Circular references disabled by default
- Child folder names must be unique per parent
- Root folder is auto-created

### Time Series Provider
- Always use timezone-aware datetime objects
- Timestamps are stored as separate value nodes
- Full precision (year through second) creates many nodes

### Time Chain Provider
- Configure `add_minutes` and `add_seconds` before creation
- Chain keys include full path (e.g., "2025:1:20:14")
- Source nodes must be created separately

### Graph RAG Provider
- LLM API calls can be cached (see `MGraph__Graph_Rag__LLM_Cache__Simple`)
- Entity confidence scores are preserved
- Relationship strengths are float values (0.0-1.0)

### RSS Provider
- Built on top of JSON provider
- Requires RSS__Feed input format
- Categories are arrays, not sets

## Conclusion

Choose providers based on:
1. **Data structure match** - Does the provider's schema align with your data?
2. **Required operations** - Do you need specific query/export capabilities?
3. **Performance needs** - How many nodes/edges will you have?
4. **Integration requirements** - Are you working with specific tools/formats?

**General Recommendation:**
- Start with **Simple** for prototyping
- Use **JSON** for API/configuration data
- Use **Mermaid** for visualization needs
- Use **Time Series** for precise temporal tracking
- Use **Graph RAG** for LLM-powered knowledge graphs

For questions or custom provider development, refer to the main MGraph-DB LLM Briefing document.
